# Event Handling

Events are the backbone of interactive web applications. An **event** is a signal that something has happened — a user click, a key press, a form submission, or the page finishing loading. JavaScript's event system lets you listen for these signals and run code in response. Understanding how events flow through the DOM, how to attach and remove handlers, and how to leverage patterns like delegation is essential for writing performant, maintainable front-end code.

---

## 1. What Is an Event?

An event is an object that represents something that happened in the browser. Every event has a **type** (e.g., `"click"`, `"keydown"`, `"submit"`) and a **target** (the DOM element where it originated).

Common event categories:

| Category   | Examples                                      |
| ---------- | --------------------------------------------- |
| Mouse      | `click`, `dblclick`, `mousedown`, `mouseup`   |
| Keyboard   | `keydown`, `keyup`                            |
| Form       | `submit`, `input`, `change`, `focus`, `blur`  |
| Document   | `DOMContentLoaded`, `load`, `scroll`, `resize`|
| Custom     | Any name you choose via `CustomEvent`         |

---

## 2. Adding Event Handlers

There are three ways to attach an event handler to a DOM element.

### 2.1 HTML Attribute (Inline Handler)

```js
<button onclick="alert('Clicked!')">Click me</button>
```

The attribute value is wrapped in a function by the browser. Discouraged because it mixes HTML with JavaScript and only allows a single handler per event type.

### 2.2 DOM Property

```js
const btn = document.getElementById("myBtn");
btn.onclick = function () {
  console.log("Clicked via DOM property");
};
```

Cleaner than HTML attributes, but still limited to **one handler** per event type. Assigning a new function overwrites the previous one:

```js
btn.onclick = handler1;
btn.onclick = handler2; // handler1 is gone
```

### 2.3 addEventListener (Recommended)

```js
const btn = document.getElementById("myBtn");

function handleClick(event) {
  console.log("Clicked!", event.type);
}

btn.addEventListener("click", handleClick);
```

**Advantages over the other approaches:**

- Supports **multiple handlers** for the same event type on the same element.
- Provides an **options object** for fine-grained control (`capture`, `once`, `passive`).
- Works with **all event types**, including custom events.

**Options parameter:**

```js
element.addEventListener("click", handler, {
  capture: false,  // listen during capturing phase (default: false)
  once: true,      // auto-remove after first invocation
  passive: true,   // promise not to call preventDefault()
});
```

The third argument can also be a boolean, which sets `capture`:

```js
element.addEventListener("click", handler, true);  // capture phase
element.addEventListener("click", handler, false); // bubble phase (default)
```

## 3. removeEventListener

To remove a handler, you must pass the **exact same function reference** and the **same capture flag**:

```js
function handleClick() {
  console.log("clicked");
}

btn.addEventListener("click", handleClick);
btn.removeEventListener("click", handleClick); // works — same reference

// This does NOT work:
btn.addEventListener("click", () => console.log("clicked"));
btn.removeEventListener("click", () => console.log("clicked"));
// Different arrow function objects — removal fails silently
```

**Key rule:** Anonymous functions cannot be removed because you have no reference to pass.

If a handler was added with `{ capture: true }`, you must pass `{ capture: true }` when removing too:

```js
btn.addEventListener("click", handler, true);
btn.removeEventListener("click", handler, true); // must match capture flag
```

## 4. The Event Object

When an event fires, the browser creates an `Event` object and passes it to every handler. Key properties:

| Property         | Description                                                          |
| ---------------- | -------------------------------------------------------------------- |
| `type`           | Event type string, e.g. `"click"`, `"keydown"`                      |
| `target`         | The element that **originated** the event (the deepest element)      |
| `currentTarget`  | The element the handler is **attached to** (always `this` in non-arrow functions) |
| `eventPhase`     | `1` = capturing, `2` = at target, `3` = bubbling                    |
| `timeStamp`      | Milliseconds since the page loaded when the event was created        |
| `isTrusted`      | `true` if generated by user action, `false` if dispatched by script  |
| `bubbles`        | Whether the event bubbles up through the DOM                         |
| `cancelable`     | Whether `preventDefault()` can cancel the event's default action     |

```js
document.addEventListener("click", function (event) {
  console.log(event.type);          // "click"
  console.log(event.target);        // the exact element clicked
  console.log(event.currentTarget); // document (where the handler is attached)
  console.log(event.isTrusted);     // true for real user clicks
});
```

## 5. Event Phases: Capturing, Target, Bubbling

When an event occurs on a deeply nested element, it travels through the DOM in three phases:

```
Phase 1 (CAPTURING): window → document → html → body → ... → parent
Phase 2 (TARGET):    → target element
Phase 3 (BUBBLING):  parent → ... → body → html → document → window
```

### Phase 1 — Capturing

The event starts at `window` and travels **down** through each ancestor to the target. Handlers registered with `{ capture: true }` fire during this phase.

### Phase 2 — Target

The event reaches the target element. All handlers on the target fire in registration order, regardless of capture flag. The `eventPhase` is `2`.

### Phase 3 — Bubbling

The event travels **back up** from the target to `window`. Handlers registered without `capture` (the default) fire during this phase.

```js
// Example: <div id="outer"><button id="inner">Click</button></div>

const outer = document.getElementById("outer");
const inner = document.getElementById("inner");

outer.addEventListener("click", () => console.log("outer capture"), true);
inner.addEventListener("click", () => console.log("inner target"));
outer.addEventListener("click", () => console.log("outer bubble"));

// Click on the button logs:
// "outer capture"   (phase 1 — capturing)
// "inner target"    (phase 2 — at target)
// "outer bubble"    (phase 3 — bubbling)
```

**Important:** Most events bubble, but some do not — `focus`, `blur`, `mouseenter`, `mouseleave`, `load`, `scroll` (on elements) do **not** bubble.

## 6. stopPropagation() vs stopImmediatePropagation()

### event.stopPropagation()

Prevents the event from traveling to the **next element** in the propagation chain. Other handlers on the **current element** still fire:

```js
inner.addEventListener("click", (e) => {
  console.log("handler 1");
  e.stopPropagation();
});
inner.addEventListener("click", () => console.log("handler 2")); // still fires
outer.addEventListener("click", () => console.log("outer"));     // does NOT fire
// Click on inner logs: "handler 1", "handler 2"
```

### event.stopImmediatePropagation()

Prevents the event from reaching **any further handlers**, even those on the **same element**:

```js
inner.addEventListener("click", (e) => {
  console.log("handler 1");
  e.stopImmediatePropagation();
});
inner.addEventListener("click", () => console.log("handler 2")); // does NOT fire
outer.addEventListener("click", () => console.log("outer"));     // does NOT fire
// Click on inner logs: "handler 1" only
```

**Best practice:** Avoid stopping propagation unless truly necessary — it can break third-party libraries or analytics that rely on events reaching `document`.

---

## 7. event.preventDefault()

Cancels the **default browser behavior** associated with an event, without stopping propagation:

```js
const link = document.querySelector("a");
link.addEventListener("click", (e) => {
  e.preventDefault(); // prevents navigation
  console.log("Link click intercepted");
});

const form = document.querySelector("form");
form.addEventListener("submit", (e) => {
  e.preventDefault(); // prevents form submission / page reload
  console.log("Form submitted via JS");
});
```

**Not all events are cancelable.** Check `event.cancelable` before calling `preventDefault()`.

`preventDefault()` is independent of `stopPropagation()`:

- `preventDefault()` → cancels default action, event still propagates.
- `stopPropagation()` → stops propagation, default action still happens.

## 8. Event Delegation

**Event delegation** is a pattern that leverages event bubbling to handle events efficiently. Instead of attaching a handler to every child element, you attach **one handler** to a common parent and inspect `event.target` to determine which child was clicked:

```js
// Without delegation: one handler per button
document.querySelectorAll(".btn").forEach((btn) => {
  btn.addEventListener("click", handleClick);
});

// With delegation: one handler on the parent
document.getElementById("toolbar").addEventListener("click", (e) => {
  if (e.target.matches(".btn")) {
    handleClick(e);
  }
});
```

**Benefits:** Memory efficient (one handler instead of many), works with dynamically added elements, simplified cleanup.

---

## 9. Delegation Pattern — target Matching and closest()

### Using matches()

```js
container.addEventListener("click", (e) => {
  if (e.target.matches("button.delete")) {
    deleteItem(e.target.dataset.id);
  }
});
```

### Using closest()

When the click lands on a nested element inside the target (e.g., an icon inside a button), `event.target` is the icon, not the button. Use `closest()` to find the nearest matching ancestor:

```js
container.addEventListener("click", (e) => {
  const btn = e.target.closest("button.delete");
  if (btn && container.contains(btn)) {
    deleteItem(btn.dataset.id);
  }
});
```

`closest()` traverses **up** from the element (including itself) and returns the first match. The `container.contains(btn)` check ensures the matched element is inside our container.

### Data Attributes for Action Dispatch

```js
const actions = { save: () => saveDocument(), delete: () => deleteDocument() };

toolbar.addEventListener("click", (e) => {
  const btn = e.target.closest("[data-action]");
  if (btn) actions[btn.dataset.action]?.();
});
```

## 10. Focus and Blur Events

| Event       | Bubbles? | When it fires                    |
| ----------- | -------- | -------------------------------- |
| `focus`     | No       | Element gains focus              |
| `blur`      | No       | Element loses focus              |
| `focusin`   | Yes      | Element gains focus (bubbles)    |
| `focusout`  | Yes      | Element loses focus (bubbles)    |

Because `focus` and `blur` do **not** bubble, event delegation requires `focusin`/`focusout` instead:

```js
form.addEventListener("focusin", (e) => { e.target.style.backgroundColor = "#ffffcc"; });
form.addEventListener("focusout", (e) => { e.target.style.backgroundColor = ""; });
```

Alternatively, use `focus`/`blur` with `{ capture: true }` since capturing works top-down:

```js
form.addEventListener("focus", handler, true);
```

## 11. Mouse Events

### Click-Related

| Event      | Description                                        |
| ---------- | -------------------------------------------------- |
| `click`    | Left mouse button press + release on same element  |
| `dblclick` | Two clicks in rapid succession                     |
| `mousedown`| Button pressed down (any button)                   |
| `mouseup`  | Button released                                    |

**Order:** `mousedown` → `mouseup` → `click` → (if double) `dblclick`

### Movement-Related

| Event          | Bubbles? | Description                                           |
| -------------- | -------- | ----------------------------------------------------- |
| `mouseover`    | Yes      | Pointer enters element **or any descendant**          |
| `mouseout`     | Yes      | Pointer leaves element **or moves to a descendant**   |
| `mouseenter`   | No       | Pointer enters element (ignores descendants)          |
| `mouseleave`   | No       | Pointer leaves element (ignores descendants)          |

`mouseenter`/`mouseleave` are often more intuitive because they don't fire when moving between child elements. However, they **do not bubble**, so you cannot use event delegation with them.

### Mouse Event Properties

Key properties available on mouse events: `button` (0=left, 1=middle, 2=right), `clientX`/`clientY` (viewport-relative), `pageX`/`pageY` (document-relative), and modifier flags `altKey`, `ctrlKey`, `shiftKey`, `metaKey`.

## 12. Keyboard Events

| Event     | When it fires                            |
| --------- | ---------------------------------------- |
| `keydown` | A key is pressed down (fires repeatedly if held) |
| `keyup`   | A key is released                        |

**Note:** The `keypress` event is deprecated. Use `keydown` instead.

### event.key vs event.code

| Property     | Description                          | Example ("A" with Shift) |
| ------------ | ------------------------------------ | ------------------------ |
| `event.key`  | The **character** produced           | `"A"`                    |
| `event.code` | The **physical key** on the keyboard | `"KeyA"`                 |

```js
document.addEventListener("keydown", (e) => {
  console.log(e.key);    // "a" or "A" (layout-dependent)
  console.log(e.code);   // "KeyA" (always the same physical key)
});
```

Use `event.key` for text input (respects keyboard layout). Use `event.code` for game controls or shortcuts (consistent across layouts).

### Modifier Keys

Combine modifier checks with key detection for shortcuts:

```js
document.addEventListener("keydown", (e) => {
  if (e.ctrlKey && e.key === "s") {
    e.preventDefault();
    saveDocument();
  }
});
```

## 13. Form Events

| Event    | Element   | When it fires                                              |
| -------- | --------- | ---------------------------------------------------------- |
| `submit` | `<form>`  | Form is submitted (Enter in input or submit button click)  |
| `input`  | Input-like | Value changes (fires on every keystroke)                  |
| `change` | Input-like | Value changes AND element loses focus (or toggle/select)  |
| `focus`  | Input-like | Element receives focus                                    |
| `blur`   | Input-like | Element loses focus                                       |
| `reset`  | `<form>`  | Form is reset                                              |

### input vs change

`input` fires on **every keystroke**. `change` fires when the value changes **and** the element loses focus (for text inputs). For `<select>` and `<input type="checkbox">`, `change` fires immediately on selection/toggle.

```js
input.addEventListener("input", (e) => console.log("input:", e.target.value));
input.addEventListener("change", (e) => console.log("change:", e.target.value));
```

### Preventing Form Submission

```js
form.addEventListener("submit", (e) => {
  e.preventDefault();
  // validate and send via fetch()
});
```

## 14. Custom Events

You can create and dispatch your own events using `CustomEvent`:

```js
// Creating a custom event
const event = new CustomEvent("user-login", {
  detail: { userId: 42, username: "alice" },
  bubbles: true,
  cancelable: true,
});

// Listening for it
document.addEventListener("user-login", (e) => {
  console.log(e.detail.username); // "alice"
});

// Dispatching it
document.dispatchEvent(event);
```

### detail Property

The `detail` property holds custom data — objects, arrays, primitives.

### Event vs CustomEvent

- `new Event("myevent")` — basic event, no `detail`.
- `new CustomEvent("myevent", { detail: data })` — includes custom data.

### Synchronous Dispatch

`dispatchEvent()` is **synchronous**. Handlers run immediately inline before `dispatchEvent()` returns:

```js
element.addEventListener("test", () => console.log("handler"));
console.log("before");
element.dispatchEvent(new Event("test"));
console.log("after");
// Logs: "before", "handler", "after"
```

## 15. Event Constructor Options

All event constructors accept an options object:

| Option       | Default | Description                                        |
| ------------ | ------- | -------------------------------------------------- |
| `bubbles`    | `false` | Whether the event bubbles up through the DOM       |
| `cancelable` | `false` | Whether `preventDefault()` has any effect          |
| `composed`   | `false` | Whether the event crosses shadow DOM boundaries    |

```js
const event = new Event("build", {
  bubbles: true,
  cancelable: true,
  composed: false,
});
```

**Important:** Custom events do NOT bubble by default. If you want delegation to work with custom events, you must set `bubbles: true`.

## 16. Scroll, Resize, and Load Events

### scroll

```js
window.addEventListener("scroll", () => {
  console.log(window.scrollY);
});
```

The `scroll` event fires very frequently. Always debounce or throttle the handler in production (e.g., using `requestAnimationFrame` to batch updates).

### resize

```js
window.addEventListener("resize", () => {
  console.log(window.innerWidth, window.innerHeight);
});
```

Like `scroll`, `resize` fires rapidly. Debounce the handler.

### load and DOMContentLoaded

```js
// DOM is ready (doesn't wait for images/styles)
document.addEventListener("DOMContentLoaded", () => console.log("DOM ready"));

// Entire page loaded (including images, styles)
window.addEventListener("load", () => console.log("Page fully loaded"));
```

**Order:** `DOMContentLoaded` fires before `load`.

---

## 17. Passive Event Listeners

Declaring a listener as **passive** tells the browser the handler will **never** call `preventDefault()`. This lets the browser optimize scrolling by not waiting for the handler to finish:

```js
document.addEventListener("touchstart", handler, { passive: true });
document.addEventListener("wheel", handler, { passive: true });
```

Without `{ passive: true }`, the browser must wait for the handler to complete before scrolling, because it doesn't know if `preventDefault()` will be called. This causes visible scroll jank.

If you call `preventDefault()` inside a passive listener, the browser ignores it and logs a warning. Modern browsers default touch and wheel listeners on `document`/`window` to passive.

---

## 18. The once Option

The `{ once: true }` option automatically removes the handler after it fires **one time**:

```js
btn.addEventListener("click", () => {
  console.log("This runs only once");
}, { once: true });
```

Equivalent to manually calling `removeEventListener` inside the handler:

```js
function handleClick() {
  console.log("This runs only once");
  btn.removeEventListener("click", handleClick);
}
btn.addEventListener("click", handleClick);
```

**Use cases:** One-time setup on first user interaction (e.g., unlock audio context), animation-end callbacks, first-click analytics tracking.

```

```
